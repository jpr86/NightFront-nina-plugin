# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

NightFront is a plugin for [N.I.N.A.](https://nighttime-imaging.eu/) (Nighttime Imaging 'N' Astronomy). A companion desktop app ("NightFront") uses forecast data and optimization to generate nightly imaging plans; this plugin ingests those plans and drives their execution inside a NINA Advanced Sequencer sequence. Target framework: `net8.0-windows` (WPF, `AssemblyName` = `JeffRidder.NINA.Nightfront`).

The plugin was scaffolded from the [nina.plugin.template](https://github.com/isbeorn/nina.plugin.template), so several files are unmodified template boilerplate rather than real NightFront features — see "Template boilerplate vs. real features" below before assuming a class is load-bearing.

## Build, test, run

- Build: `dotnet build NightFront.slnx` (or open in Visual Studio; solution is a `.slnx`, not `.sln`).
- Test: `dotnet test NightFront.Tests/NightFront.Tests.csproj` (xUnit + Moq). Run a single test with `dotnet test --filter "FullyQualifiedName~NightFrontJsonImporterTests.MethodName"`.
- Note: some importer tests that build a real `InputTarget`/`InputCoordinates` graph transitively call into NINA.Astrometry's native `NOVAS31lib.dll`. That DLL isn't in the NuGet package — it only ships with a real NINA install — so those specific tests only pass on a machine where it's been copied into the test project's output folder. The rest of the importer tests avoid astrometry code and run anywhere.
- Post-build step: `NightFront.csproj` has a `PostBuild` target that xcopies the build output into `%localappdata%\NINA\Plugins\3.0.0\<TargetName>` so a local NINA install picks up the plugin immediately — this only works on Windows with NINA installed. That same target **then** runs `BuildNativeCli.bat` (`gradlew nativeCompile`), a multi-minute GraalVM native-image build of `nightfront-cli.exe` that fires on **every** `dotnet build` (a `java.exe` at several hundred MB is the tell; see CLI deployment below). It never fails the plugin build (warns and continues), and the C# compile is already done before it starts — so to iterate on tests, build once and then `dotnet test … --no-build`, or kill that `java.exe` to let a build finish immediately.
- `CopyLocalLockFileAssemblies` is `true` deliberately: NINA loads this plugin standalone (not via a referencing project), so any NuGet package it references directly (e.g. Ookii.Dialogs.Wpf) must be copied alongside the built DLL. Conversely, `NINA.Plugin`'s own package reference uses `ExcludeAssets="runtime;native"` — NINA's own assemblies must come from the host process at runtime, not a private copy, or types like `ISequenceItem` fail to cast across the plugin/host boundary.

## Architecture

**Data flow:** the external NightFront app writes a nightly plan as a JSON file (a hand-rolled mimic of NINA's own sequence-serialization shape) into a configured folder. Inside NINA, the user builds a sequence containing a "Nightly Update" instruction (`NightFrontUpdateInstruction`) followed later in the same sequence branch by a "NightFront Container" (`NightFrontContainer`) — the Container need not be an immediate sibling; it may be nested inside a later sibling's descendant containers (e.g. a container that loops repeatedly until dawn), as long as Update itself sits in a container that runs once, near the start of the night. The update instruction runs once, finds the container reachable by scanning forward, and populates it. At runtime:

1. `NightFrontUpdateInstruction.Execute` (`Sequencer/NightFrontUpdateInstruction.cs`) looks in `Settings.Default.NightFrontDataFolder` for a `*.json` file whose name contains today's date (`yyyy-MM-dd`), and updates its own `LastRunSucceeded`/`StatusMessage`/`LastRunTimestamp` properties (bound in the sequencer UI as a status dot + message) regardless of outcome.
2. It hands the file contents to `NightFrontJsonImporter.Import` (`Import/NightFrontJsonImporter.cs`), which walks the JSON tree and constructs real, live `NINA.Sequencer` objects (`DeepSkyObjectContainer`, `SequentialContainer`, `SwitchFilter`, `TakeExposure`, `Dither`, `RunAutofocus`, `StartGuiding`, `CenterAndRotate`, `TimeCondition`, `LoopCondition`) — this is the closed set of instruction types the importer can parse. `RunAutofocus` is kept in this list for backward compatibility with old plan files only — NightFrontApp's exporter no longer emits it (removed per explicit request; refocusing is left entirely to the user's own sequence-level autofocus triggers), so "can parse" and "NightFront currently emits" are no longer the same set. Unrecognized `$type` values throw `NightFrontImportException`. The importer does not guarantee `DeepSkyObjectContainer` at the top level specifically — `BuildItem` accepts any of the supported types there, by convention rather than an enforced constraint.
3. `NightFrontUpdateInstruction.Execute` locates the target container via `NightFrontContainer.FindNext(Parent.Items, this)` (a forward scan of its own parent's sibling list that also descends depth-first into each following sibling's subtree, so the container can be nested arbitrarily deep) and calls `NightFrontContainer.PopulateItems` to replace the container's children with the imported items, which then execute like any normal part of the sequence natively — the plugin does not interpret or manage execution itself (no custom pause/resume engine).
4. On success, it also creates a `NightFrontMetadataRecorder` (`Import/NightFrontMetadataRecorder.cs`), which finds each target's `CenterAndRotate` instruction and subscribes to its `PropertyChanged`. When a target's `CenterAndRotate` finishes, the recorder reads the *measured* rotator position from `IRotatorMediator.GetInfo().MechanicalPosition` — deliberately not the plan's input Sky PA (`CenterAndRotate.PositionAngle`/`Target.PositionAngle`), since calibration frames need to match the physical rotator position a flat/dark frame will actually be shot at — pairs it with that target's (statically known) filters, and incrementally records each into `calibration.metadata.json` — a single, fixed-name, **undated** file in the data folder (path from `NightFrontMetadataPaths.GetLiveMetadataPath(folder)`; all reads/writes go through `NightFrontMetadataStore`) that accumulates calibration requirements across nights. The name is deliberately fixed, not derived from the plan file or the date: an earlier date-stripped scheme forked a second, disconnected metadata file whenever a writer (Update/Replan) ran after local midnight, because the stripped-out "today" token no longer matched the plan's own unchanging night date. Filter/rotation-angle pairs are deduped within 1 degree. Best-effort throughout — a write failure, or a target whose rotation never gets measured (e.g. the sequence stops early), doesn't fail the import.

Why a hand-rolled importer instead of NINA's native JSON sequence deserializer: `SequenceJsonConverter`/`TemplateController` are internal to NINA's composition root and unavailable to plugins. The [tcpalmer/nina.plugin.targetscheduler](https://github.com/tcpalmer/nina.plugin.targetscheduler) plugin takes the same approach for its own externally-sourced plan data and is a useful reference point.

**Calibration flats (consuming the metadata).** The `calibration.metadata.json` the recorder accumulates is consumed by a set of calibration instructions/conditions that shoot matching flats at each measured rotation angle: `NightFrontSkyFlatsInstruction`/`NightFrontTrainedFlatsInstruction` (rotate to, then shoot flats for, the next outstanding requirement per `Settings.Default.FlatFilterOrder`, stamping `FlatsCompletedDate` on success or releasing the claim on failure), `NightFrontRotateToNextAngleInstruction` (rotate to the next requirement's angle without consuming it), and the `NightFrontWhileCalibrationRemainsCondition`/`NightFrontWhileSameRotationCondition` loop guards. All resolve the file via `NightFrontMetadataPaths.ResolveExistingMetadataPath`, and when it doesn't exist yet (no imaging has run) they exit / loop-false with an informational message rather than failing the sequence. `NightFrontMetadataStore` centralizes every read/modify/write under one process-wide lock; a completed requirement stays in the file with `FlatsCompletedDate` stamped and is removed only by `PruneStaleCompleted` once older than `Settings.Default.FlatRefreshDays` (at which point the recorder re-adds it as fresh if that filter/angle/gain/offset is imaged again).

**Safety-recovery replan (`Sequencer/NightFrontReplanInstruction.cs`, todos/nina-safety-delay-plan.md Phase 3):** a second sequence instruction, "Replan" (renamed from "Replan After Safety Recovery" once it became clear it isn't actually tied to that one branch — see the CLI-deployment paragraph below and the plan doc's post-Phase-5 notes), meant to be placed by the user inside (or immediately after) their own template's `Once Safe` branch — the point a safety-monitor interruption's recovery handling already falls back to, right before control returns to the top of `Loop while safe`. Unlike `NightFrontUpdateInstruction`, it has no fixed positional relationship to the `NightFrontContainer` it needs to reach (`Once Safe` and `Loop while safe`, where the container actually lives, are sibling branches, not a preceding-sibling arrangement), so it locates the container via `NightFrontContainer.FindAnywhere(this)` — walks up to the ultimate root container, then searches every descendant depth-first — rather than `FindNext`'s forward-from-a-specific-sibling scan. On execution it:
1. Reads `NightFrontContainer.BuildProgressSnapshot()` (Phase 1) and writes it to the `.progress.json` sidecar for tonight's plan file (`NightFrontMetadataPaths.GetProgressSnapshotPath`).
2. Reads live cloud cover / connectivity via `IWeatherDataMediator.GetInfo()` (the same `GetInfo()`-based mediator pattern `NightFrontMetadataRecorder` already uses for `IRotatorMediator`) and, if connected, writes a `live-weather-override.json` file in NightFront's `LiveWeatherOverride` JSON shape; otherwise passes the literal string `"none"`.
3. Spawns the CLI (resolved via `NightFrontMetadataPaths.ResolveCliPath` — see the CLI-deployment paragraph below; usually the co-located `nightfront-cli.exe`, not `Settings.Default.NightFrontCliPath` directly) with `replan --effort=Fast <session-config.json> <progress-snapshot> <weather-override|none> <output-plan> [selection-preference.json]` (the effort is the hardcoded `NightFrontReplanInstruction.ReplanEffortLevel` constant, no longer a setting — see Settings below; assembled by `NightFrontReplanInstruction.BuildReplanArguments`, a small pure/testable helper) and waits up to `ReplanTimeoutSeconds` (300s) for it to exit. `<output-plan>` is a fresh, GUID-suffixed temp path (`BuildCliOutputPath`), deliberately **not** the plan file NightFrontUpdateInstruction would look for — that file routinely already exists on disk (tonight's already-imported plan), so checking its mere existence afterward couldn't distinguish "the CLI wrote something new" from "this was already there" (a real bug caught by `/code-review` and fixed — see the plan doc's Phase 3 status).
4. On success, imports the temp output plan the same way `NightFrontUpdateInstruction` does and calls the existing `NightFrontContainer.PopulateItems` to fully repopulate the container — safe here specifically because nothing in `Loop while safe` is executing at this point. It then archives a timestamped copy of the previous plan file into a `replan-history/` subfolder (`ArchivePreviousPlanFileIfPresent`/`NightFrontMetadataPaths.GetReplanHistoryPath` — a subfolder specifically so `FindTodaysPlanFile`'s non-recursive scan never mistakes an archived copy for today's real plan, with no explicit exclusion rule needed) before `File.Move`-ing the temp path over the real plan file, which also makes that on-disk update atomic. If the CLI exits 0 but writes no temp file at all (NightFront's own replan mode does this exactly when every target is already fully imaged — see NightFrontApp's `Main.kt` `runReplan`), the container is instead emptied, so the native restart-from-top doesn't re-shoot a finished night.

This requires NightFrontApp's own exporter to have also written two new sidecars this instruction depends on that `NightFrontUpdateInstruction`'s existing plan-file scan never needed: `session-config.json` (NightFront's own input config for tonight's plan — the plugin otherwise has no way to know where it lives, since it only ever reads the already-transformed NINA sequence JSON) and `selection.json` (the operator's originally-chosen utilization/quality tradeoff, optional). Both are single, undated files per export directory (not per-night like the progress snapshot), and both are excluded from `NightFrontMetadataPaths.FindTodaysPlanFile`'s "what's the actual plan file" scan alongside the existing `.metadata.json`/`.progress.json` exclusions.

**CLI deployment.** The NightFront CLI is built as a GraalVM native-image executable (`nightfront-cli.exe`, targeting `MainKt` — NightFrontApp's `build.gradle.kts`, `graalvmNative` block) and copied alongside this plugin's own DLL by `NightFront.csproj`'s `PostBuild` target (`BuildNativeCli.bat`), which shells out to `..\gradlew.bat nativeCompile` — relies on `nina-plugin` always being checked out as a submodule inside `nightfrontapp`, and never fails the plugin build if that sibling checkout is missing or the native build fails (warns and continues; `NightFrontCliPath` remains a manual fallback). `NightFrontMetadataPaths.ResolveCliPath` (moved here from `NightFrontReplanInstruction` since it's the same "where does the plugin find a file it depends on" logic that class already centralizes for every other sidecar) prefers `Settings.Default.NightFrontCliPath` only when it's set *and* points at a file that actually exists; otherwise it resolves `nightfront-cli.exe` co-located with the plugin's own assembly directory — the common case needs zero configuration, and `NightFrontCliPath` (no longer exposed on the Options tab) exists only as a hand-editable `user.config` escape hatch. Log4j2's reflective `StatusLogger` init (and other runtime reflection) needed real GraalVM tracing-agent-captured config, not hand-authored reflect-config — see `regenerateNativeImageConfig` in NightFrontApp's `build.gradle.kts` and the committed metadata at `src/main/resources/META-INF/native-image/com.ridderware/nightfront/` in that repo.

**Validation constraint:** `NightFrontUpdateInstruction.Validate()` enforces that a `NightFrontContainer` is reachable by scanning forward through its later siblings and their descendants in the same parent (via `NightFrontContainer.FindNext`, not a parent/child relationship to Update itself) and that the data folder setting is configured — both are structural requirements the sequencer UI can't otherwise guarantee. The class name `NightFrontUpdateInstruction` is intentionally unchanged from its "NightFront Update" display name (now "Nightly Update") to avoid breaking `$type`-based deserialization of previously-saved sequences.

**Container summary view:** `NightFrontContainer.PopulateItems` also rebuilds `TargetSummaries`, an `ObservableCollection<object>` of `NightFrontTargetSummary` (one per top-level `DeepSkyObjectContainer`) or the raw `ISequenceItem` for any other top-level type, rendered via `NightFrontTargetRowTemplateSelector` in the sequencer UI. Each `NightFrontTargetSummary` (`Sequencer/NightFrontTargetSummary.cs`) is a live view of one target: name/coordinates/rotation/window-end are snapshotted once from the imported plan, `WindowStart` is chained from the previous target's `WindowEnd` (a target's own subtree only carries its end-of-window `TimeCondition`, not a start), and `CompletedCount`/`Status` stay live by subscribing to the underlying `LoopCondition`/`TakeExposure`/container `PropertyChanged` events as NINA executes.

**Settings:** `Settings.Default.*` (`Properties/Settings.Settings`, generated `Settings.Designer.cs`) — global, not per-equipment-profile. `NightFrontDataFolder` lives here because a NightFront installation exports to one location regardless of which NINA equipment profile is active. `NightFrontCliPath` (empty by default — optional; only needed as a manual override when the co-located `nightfront-cli.exe` build is missing, not exposed on the Options tab) was added for the Phase 3 replan feature above. The replan GA compute budget is **not** a setting: `NightFrontReplanInstruction` hardcodes a `ReplanEffortLevel = "Fast"` constant that's passed to the CLI verbatim as `--effort=Fast`, where NightFront resolves it against its own **`ReplanEffortPreset`** (`optimizer/ReplanEffortPreset.kt`) — a replan-specific preset family deliberately much smaller than the desktop app's GUI `EffortPreset` (replan "Fast" = pop 60 / gen 50, vs the GUI Fast's pop 100 / gen 200), because a replan only re-solves the Active-target remainder of one night, yielding a ~2s solve on typical imaging-PC hardware. It **used to** be a Fast/Balanced/Thorough Options-tab dropdown (`Settings.Default.ReplanEffortLevel` plus the bound `ReplanEffortLevel`/`ReplanEffortLevelOptions` properties on `Nightfront.cs`), but since `ReplanEffortPreset` was already sweep-tuned so its Fast tier is exactly right for the unattended imaging-PC case, a user-overridable dropdown only threatened to undo that tuning — so the dropdown, both properties, and the setting were removed in favor of the hardcoded constant. The plugin passes the name only and is agnostic to the pop/gen behind it, so this coupling lives entirely on the NightFront (CLI) side. (`Nightfront.cs` also holds an unused `IProfileService` reference, kept as a standard hook for any future profile-specific feature — the plugin template's original per-profile settings demo, `IPluginOptionsAccessor`/`ProfileSpecificNotificationMessage`, was removed as vestigial.)

**Target start/end broadcast (`Sequencer/NightFrontTargetTriggerBase.cs`, `NightFrontBeforeTargetTrigger.cs`, `NightFrontAfterTargetTrigger.cs`; todos/nina-target-broadcast-plan.md):** two more `SequenceTrigger`s, near-clones of `NightFrontSeeingTrigger` minus its poller, that fire around each imported target's own `DeepSkyObjectContainer` and run whatever instructions the user drops into their own `TriggerRunner` — typically a Ground Station send. "Before Target" fires from `ShouldTrigger` when `nextItem` is that target's container; "After Target" fires from `ShouldTriggerAfter` when `previousItem` is (this also fires for the night's last target, since NINA's own `SequentialStrategy` calls `RunTriggersAfter` once more with `nextItem == null` after the final item). Both share `NightFrontTargetTriggerBase.IsNightFrontTarget` — `item is DeepSkyObjectContainer dso && dso.Parent is NightFrontContainer` — which is load-bearing, not defensive: because triggers propagate up the parent chain, a trigger on an ancestor (e.g. "Loop while safe") also receives every inner item pair from every nesting level in the whole tree below it, including unrelated `DeepSkyObjectContainer`s in the user's own template. The reason this makes Ground Station's `$$TARGET_NAME$$`/`$$TARGET_RA$$`/`$$TARGET_DEC$$` resolve: `Utilities.FindDsoInfo` (`daleghent/nina-ground-station`) walks only *up* the parent chain for the nearest `IDeepSkyObjectContainer`, but `TriggerRunner`'s own parent is never set by NINA (`SequenceContainer.Add(ISequenceTrigger)` attaches only the trigger, not its runner) — so `Execute` re-parents `TriggerRunner` onto the firing target's container for the duration of the run, then restores the original parent in a `finally` (never leaving it pointed at an imported container a replan/Nightly-Update can replace out from under it, and never letting that dangling reference get serialized — `TriggerRunner` is `[JsonProperty]`). `Validate()` checks both that a `NightFrontContainer` is reachable at all (`NightFrontContainer.FindAnywhere`-equivalent logic, since a trigger is an `ISequenceTrigger` not an `ISequenceItem` and can't call `FindAnywhere` directly) **and** that it's actually a descendant of this trigger's own `Parent` — catching the "Once Safe" misplacement (a sibling branch of "Loop while safe" that never sees the container's targets execute) that a plain reachability check would miss.

**MEF composition:** NINA discovers plugin pieces via `System.ComponentModel.Composition` exports, not DI registration you'll find by searching for a container. Look for `[Export(typeof(...))]` / `[ImportingConstructor]` to find entry points: `Nightfront` (`IPluginManifest`, in `Nightfront.cs`) is the plugin root and options-tab data context; `NightFrontUpdateInstruction`/`NightFrontReplanInstruction` export `ISequenceItem` and carry `[ExportMetadata(...)]` for their sequencer-palette name/icon/category. `NightFrontContainer` exports **both** `ISequenceItem` and `ISequenceContainer` - matching how NINA's own container types (e.g. `DeepSkyObjectContainer`) are exported - because NINA resolves containers during sequence deserialization from a separate `ISequenceContainer` export list (`NINA.Sequencer.SequencerFactory.GetContainer<T>()`), distinct from the `ISequenceItem` list used for the palette/drag-drop. Missing the `ISequenceContainer` export made a saved-and-reopened sequence show it as an "Unknown Instruction" placeholder (fixed in 1.0.0.5).

## Template boilerplate vs. real features

The plugin template's sample code has been removed as unused boilerplate, since none of it was part of NightFront's actual plan-import feature: `NightfrontDockables/`'s altitude-chart panel, `NightfrontDrivers/`'s fake random-data weather device/provider, and `Nightfront.cs`'s example FITS-keyword injection and example image pattern (`exampleImagePattern`, the `ImageSaveMediator_*` handlers).

## Known open work

The items in `todos/todos.txt` have been implemented as they were added — see `CHANGELOG.md` under 1.0.0.2 (sibling placement + status, the palette icon, the container summary, calibration metadata), 1.0.0.3 (removing vestigial options-tab settings, the remaining container icon, the plugin-list icon, and switching calibration metadata to measured rather than planned rotation angles), and 1.0.0.4 (allowing the NightFront Container to be nested inside descendant containers, such as a nightly imaging loop, instead of requiring it to be Update's immediate sibling). The one question `todos/todos.txt` left explicitly open, whether the plugin should manage execution itself (pause/resume on unsafe conditions, like `tcpalmer/nina.plugin.targetscheduler`) instead of letting NINA execute the imported plan natively, was decided in favor of native execution for now. Treat `todos/todos.txt` as the live source of truth for pending direction — it gets overwritten with new items as the maintainer adds them, so don't assume its current contents match what's described here by the time you read this.
